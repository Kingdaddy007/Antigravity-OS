# Brand Translation and Handoff

## ENTRY GATE

Require an approved diagnosis and creative brief or equivalent constraints. Confirm:

- desired perception and first-known priority;
- audience and wrong-fit audience;
- controlling problem and proof burden;
- taste world and rejected category convention;
- real assets and claims;
- accessibility, responsive, performance, maintenance, and authority constraints.

Without these inputs, return source findings only. Do not make adoption decisions.

## TWO-PASS MODEL

### Pass 1 — Pre-Territory Vocabulary

Run after the creative brief. Use references to expand candidate vocabulary, identify proven communication patterns, expose risks, and create a bounded pattern ledger. Do not select the page concept or assign every candidate to a section.

### Pass 2 — Territory Validation

Run after structurally different territories and rough tests exist. Use references to answer precise questions about continuity, hierarchy, proof, interaction, motion, responsive translation, assets, and feasibility. Reject any territory that depends on reference prestige or copied architecture.

## DECISION LEDGER

Classify every material pattern:

| Decision | Meaning |
| --- | --- |
| `KEEP` | Existing approved behavior is validated and should remain substantially unchanged |
| `ADAPT` | Retain the principle while changing form, scale, intensity, trigger, ownership, content, or implementation |
| `REJECT` | Conflict with brand truth, proof, usability, assets, performance, maintenance, or narrative hierarchy |
| `DEFER` | Potentially useful after a route, asset, integration, or product condition exists |

Use this record:

| Field | Required Content |
| --- | --- |
| Candidate | Neutral name for the mechanic |
| Decision | Keep, Adapt, Reject, or Defer |
| Communication job | What understanding or action it improves |
| Strategic source | Source IDs and confidence |
| Retained principle | The transferable logic |
| Changed form | Geometry, intensity, trigger, duration, content, and ownership |
| Evidence / assets | Claims, imagery, video, plans, copy, or interaction states required |
| Static comparison | Why stillness is insufficient, or why stillness wins |
| Mobile equivalent | Complete touch/narrow-layout behavior |
| Reduced-motion equivalent | Complete static or low-travel behavior |
| Cost / risk | Implementation, performance, accessibility, maintenance, and content burden |
| Rejection condition | What evidence or failure would reverse approval |

## BRAND FILTER

Test every candidate in this order:

1. Does it strengthen the approved positioning rather than borrow another brand's identity?
2. Does it serve a named communication job in the controlling argument?
3. Can real evidence and assets support it?
4. Is the retained principle stronger than a still composition?
5. Does it preserve proof, navigation, inquiry, and reading intervals?
6. Does mobile preserve meaning without imitating desktop travel?
7. Does reduced motion preserve the complete argument?
8. Is the cost proportional to the chapter's importance?
9. Can the team maintain it with the actual stack and capacity?
10. Does the page remain coherent if the reference name and prestige disappear?

Reject on the first load-bearing failure. Do not compensate with polish.

## IMPLEMENTATION HANDOFF LEVELS

Choose only the level authorized and needed:

### Concept Input

Provide research questions, candidate principles, negative patterns, asset burdens, and rejection conditions. Leave territory invention open.

### Experience Input

Map approved patterns to chapter jobs, proof timing, continuity, interaction, responsive intent, and motion ownership.

### Production Input

Define assets, text-safe zones, behavior states, timeline responsibility, fallbacks, performance constraints, build order, and acceptance criteria.

### Section Plan Set

Create a master sequence, shared foundation, and focused section plans when multiple chapters have independent owners or risks. Each plan must contain:

- approved outcome and stable communication job;
- observed reference evidence;
- brand-specific adaptation;
- implementation inference;
- current-state changes;
- content and asset contract;
- motion job and stillness comparison;
- desktop, mobile, reduced-motion, loading, and failure behavior;
- accessibility and performance constraints;
- build steps and acceptance criteria.

Do not force a small site into multiple files. Use an equivalent section inside approved concept or production documents when that is clearer.

## FINAL INTEGRITY TRACE

Trace:

`brand evidence → creative criteria → reference question → source observation → corpus pattern → translation decision → territory or chapter job → asset and fallback → implementation owner → acceptance test`

Stop the handoff when any arrow depends on an unsupported claim.
